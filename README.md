# Air Filter Shipment System

## Setup

**Prerequisites:** Node.js, `sqlite3` CLI on PATH.

```bash
npm install
bash reset.sh        # builds DB, seeds fixtures and properties
```

Then in two terminals:

```bash
npm run dev:backend  # Express on :3000
npm run dev:frontend # Vite on :5173 ‚Äî open http://localhost:5173
```

> `reset.sh` is destructive ‚Äî it drops and rebuilds `database.db` from scratch. Run it once on first setup or when you need a clean slate.

---

## Data Model

The system extends the pre-existing `tenants`, `enrollments`, and `historical_shipments` tables with:

- **`properties`** ‚Äî one row per property slug from `properties.json`. `id` is `TEXT` (slugs like `"prop-cedar-court"`). `address` is nullable ‚Äî `properties.json` provides none.
- **`property_tenants`** ‚Äî junction table linking tenants to properties. Needed because the fixture data has tenant 135452 in both `prop-oak-terrace` and `prop-oak-overflow`; a junction table handles this without touching the immutable `tenants` table.
- **`shipments`** ‚Äî exports generated by this system. Kept separate from `historical_shipments` so eligibility queries are unambiguous ‚Äî only rows in `shipments` count toward the double-ship guard.
- **`historical_shipments.custom_field_1`** ‚Äî added via `ALTER TABLE` (non-destructive). Stores filter sizes on both legacy records and ShipStation imports.

## Assumptions & Decisions

### Eligibility

- **"Today" is 2026-04-24.** Any `ship_date <= 2026-04-24` counts as a received shipment and resets the interval clock. Future-dated records are ignored (pending/unconfirmed). Exported-but-untracked shipments still block re-export ‚Äî an order is enough to disqualify.
- **Air-filter rider matching** normalizes both `"Free Airfilters Delivery"` and `"Airfilters Delivery ($4)"` by lowercasing and checking for the substring `"airfilter"`.
- **Multi-property tie-break:** when a tenant belongs to multiple properties with different intervals, use the longest interval. Safer to under-ship than over-ship.
- **Tenants with no property assignment** (36 in the DB) are excluded ‚Äî no property means no interval to evaluate against.

### Properties Seeding

Seeding runs as a separate step (`npm run seed-properties`) after fixtures, not baked into the migration. Reason: `prop-maple-loop` references tenants 900001/900002, which only exist after `seed-fixtures` runs.

`INSERT OR IGNORE` on both `properties` and `property_tenants` keeps the script idempotent.

| Issue in `properties.json` | Decision |
|---|---|
| `prop-riverbend` appears twice (90-day "Riverbend" + 45-day "Riverbend Annex") | First-wins on the primary key. Annex skipped; tenant 145566 stays on 90-day interval. |
| `prop-oak-overflow` lists tenant 135452, already in `prop-oak-terrace` | Both links kept. Eligibility uses longest interval (120 days). |
| Tenant IDs not present in `tenants` | Skipped with a warning. Only affects `prop-maple-loop` if fixtures haven't run. |

### ShipStation Import Matching

Automatic match requires both name and address to agree (case-insensitive, trimmed). Either alone is not treated as confident ‚Äî address reuse across tenants and name typos are both common enough to warrant manual review over a bad auto-match.

### `shipments.tracking_number`

Currently `NOT NULL UNIQUE`, but at export time only the order exists ‚Äî tracking comes back from ShipStation later. This column should be nullable; `NULL` means "ordered, not yet confirmed." Left as-is to avoid breaking import dedup logic mid-implementation; the fix is straightforward.

### Data State

`schema.sql` contains 1,258 commented-out `INSERT INTO enrollments` rows ‚Äî the apparent original dataset. The current DB has only the enrollments from `supplemental-fixtures.sql`, so most tenants are ineligible under the enrollment check. If the original `database.db` was overwritten by `reset.sh`, those enrollments were lost. The eligibility engine is correct against the schema as-is; restoring the original data would surface the full eligible set.

## Dependencies

| Package | Why |
|---|---|
| `better-sqlite3` | Synchronous SQLite driver ‚Äî fits Express's synchronous route handlers; standard choice for single-process Node servers. |
| `csv-parse` | Parses ShipStation CSV uploads in `importService.js` via the `sync` entrypoint. |
| `express` | HTTP server and routing. |
| `multer` | Multipart form handling for CSV uploads ‚Äî `express.json()` doesn't handle `multipart/form-data`. |
| `react` / `react-dom` | Frontend UI. |
| `sonner` | Toast notifications ‚Äî handles positioning, stacking, dismiss, and colors out of the box. |
| `vite` + `@vitejs/plugin-react` | Frontend bundler and dev server with React Fast Refresh. |
| `nodemon` | Restarts Express on file changes during development. |
