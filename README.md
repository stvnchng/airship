# Air Filter Shipment System

## Setup

**Prerequisites:** Node.js, `sqlite3` CLI on PATH.

```bash
npm install
bash reset.sh        # builds DB, seeds fixtures and properties
```

Then in two terminals:

```bash
npm run dev:backend  # Express on :3000
npm run dev:frontend # Vite on :5173 (open http://localhost:5173)
```

> `reset.sh` is destructive. It drops and rebuilds `database.db` from scratch. Run it once on first setup or whenever you need a clean slate.

## Data Model

The system extends the pre-existing `tenants`, `enrollments`, and `historical_shipments` tables with new tables, and adds two columns to `historical_shipments`.

### `properties`

One row per property from `properties.json`. `id` is a text slug (e.g. `"prop-cedar-court"`). The key field is `shipment_interval_days`: how many days must pass between shipments for tenants at that property. `address` is nullable; `properties.json` provides none.

### `tenant_property`

Maps each tenant to a property so the eligibility engine can look up the shipment interval. The spec defines a tenant as belonging to one property, so the table carries a `UNIQUE tenant_id` constraint enforcing that. The junction shape (rather than a direct foreign key on `tenants`) was chosen to avoid altering the immutable `tenants` table.

### `shipments`

Records shipments *generated by this tool*. The spec allows using `historical_shipments` directly or building a separate model. A separate table is the right call here because `historical_shipments.tracking_number` is `NOT NULL UNIQUE` in the original schema, and this tool generates export records before tracking numbers exist. Merging would require either relaxing that constraint or inserting placeholder values, both of which are worse than a clean separate table.

Keeping them separate also makes the double-ship guard unambiguous. The eligibility query UNIONs both tables in `recently_shipped`: a tenant is blocked by either a confirmed past delivery or a pending export order. `tracking_number` on `shipments` is nullable because tracking comes back from ShipStation after fulfillment.

### `historical_shipments` (extended)

- **`custom_field_1`:** filter size, populated by ShipStation CSV import. Used as `last_filter_size` on the eligibility list and written into the export CSV.
- **`review_reason`:** nullable. `NULL` means the row is clean; a non-null value (`'unmatched'`, `'ambiguous_match'`, `'no_filter_size'`) puts it in the manual review queue. Cleared back to `NULL` when the operator resolves or dismisses it. This keeps the queue as a filtered view of `historical_shipments` rather than a separate table, which simplifies queries, but it does mean import state lives on the shipment record itself.

### `import_log`

Records each actual CSV upload. The export gate needs to know whether any real import has happened. The seeded fixture data carries historical ship dates, so inferring "has imported" from `MAX(ship_date)` would always return true and the gate would never fire. `import_log` makes the signal explicit.

### `export_batches`

Stores the full generated CSV alongside metadata (date, `as_of`, tenant count). Supports re-download via `GET /api/export/last`. See **Export commit before CSV delivery** under Assumptions & Decisions for why this exists.

## Data Observations

Things we noticed about the provided data that directly informed implementation decisions.

**`enrollments.riders` is not normalized.** The column stores a PostgreSQL array-style string (`{val1,val2,...}`) from the original schema. The air-filter rider appears under two labels (`"Free Airfilters Delivery"` and `"Airfilters Delivery ($4)"`), and both can appear in the same row. Matching requires `LIKE '%Airfilters Delivery%'` rather than an equality check.

**ShipStation CSV zip codes are inconsistent.** The provided `shipstation-export.csv` has three zip formats in the same file: 4-digit truncated (e.g. `3095` instead of `03095`), standard 5-digit, and ZIP+4 (e.g. `92328-6076`). Tenant records in the DB use 5-digit or ZIP+4. `normalizeZip()` handles all three by stripping non-digits, taking the first 5 characters, and left-padding to 5. SQL compares against `substr(zip, 1, 5)`.

**`shipstation-export.csv` uses `shipment_id`, not `tracking_number`.** The column used as the dedup key is ShipStation's internal shipment ID (`se-216966616` style), not a carrier tracking number. It's stable and unique within ShipStation so it works for dedup, but it's not the UPS/FedEx tracking number a tenant would look up.

**`custom_field_1` can contain multiple space-separated sizes.** 9 of 195 rows in the provided CSV have values like `"14x20x1 20x20x1"`. These are stored verbatim. The spec notes that sizes "may lack the structure that a real API would require," and storing verbatim passes the value through to the export CSV unchanged.

**`properties.json` has two data quality issues.** The property ID `prop-riverbend` appears twice with different intervals (90 days and 45 days). Two tenants are also listed more than once: tenant 135452 is in both `prop-oak-terrace` and `prop-oak-overflow`; tenant 145566 is in both `prop-riverbend` entries. Both issues need a defined fallback. See the Loading `properties.json` section below.

**Two tenants share identical name and address.** Tenants 900001 and 900002 are both named Casey Morgan at `1188 Maple Loop, Apt. 4B, Springfield, IL 62704`. The only differentiator is their ID. This is exactly what makes `ambiguous_match` necessary: name+zip matching correctly identifies the conflict but cannot pick the right tenant automatically.

**36 tenants have no property assignment.** 126 tenants are in the DB; only 90 appear in `properties.json`. The 36 without a property assignment have no interval to evaluate against, so they are excluded from eligibility entirely.

**Most enrollments are inactive.** `schema.sql` contains 1,258 enrollment rows, but 1,044 have `active = false`. Only 214 are active. All rows are loaded into the DB by `reset.sh`, so most tenants appear ineligible because their enrollment is inactive, not because data is missing.

**The fixture data spans dates both before and after "today."** `historical_shipments` is pre-seeded with ship dates ranging from `2025-10-15` to `2026-05-15`. The future-dated record (`2026-05-15`, source `scheduled_order`) is beyond the hardcoded `asOf` of `2026-04-24` and is correctly excluded by `WHERE ship_date <= @asOf`. The pre-seeded historical records also mean `MAX(ship_date)` cannot reliably signal whether a real import has occurred, which is why `import_log` exists.

## Assumptions & Decisions

### Eligibility

- **"Today" is 2026-04-24.** Shipments within a tenant's property interval and on or before this date count toward the cooldown. Future-dated records are ignored. Tenants who have never received a shipment are eligible.
- **Exported-but-unconfirmed shipments count.** The spec asks for a decision here. Rows in `shipments` block re-export within the interval window because an order is enough to disqualify. It's the safer default against double-shipping.
- **Air-filter rider matching** uses `LIKE '%Airfilters Delivery%'` against the `riders` text blob. Matching on a shared substring means future rider label variants won't require a code change. It also works around the fact that `riders` is a PostgreSQL array-style string from the original schema and can't be queried cleanly any other way without normalizing the column first.
- **Tenants with no property assignment** (36 in the DB) are excluded. No property means no interval to evaluate against.

### Loading `properties.json`

`properties.json` is the source of property intervals and tenant assignments per the spec. The file has a few conflicts that each need a defined fallback:

| Conflict | Decision |
|---|---|
| Two entries share the same property ID | First entry wins; second is skipped with a warning. |
| One tenant is listed under two different properties | First assignment wins; second is skipped with a warning. |

The script logs a warning rather than failing, so setup completes with the remaining clean data intact.

### ShipStation Import Matching & Manual Review

The spec says to match on name, address, or a combination. We use **name + zip** as the match key. Full address matching is brittle because unit formatting varies and abbreviations differ. Zip is the most stable, structured part of an address. Either field alone has too many collisions to be a confident match.

A row is flagged for manual review under three conditions:

| Reason | What it means | Required action |
|---|---|---|
| `unmatched` | No tenant found with the same name and zip | Pick the correct tenant from the suggested candidates, or dismiss |
| `ambiguous_match` | Two or more tenants share the same name and zip | Confirm which tenant this shipment belongs to, or dismiss |
| `no_filter_size` | Matched confidently, but `custom_field_1` is blank | Enter the filter size if known, or dismiss to skip (the spec says size issues should not block processing) |

**Zip normalization:** ShipStation exports have inconsistent zip formats in the same file. `normalizeZip()` strips non-digits and left-pads to 5 characters. SQL compares against `substr(zip, 1, 5)`.

### Filter sizes

`custom_field_1` cells may contain multiple space-separated sizes (e.g. `"14x20x1 20x20x1"`). They are stored verbatim, per the spec's note that sizes "may lack the structure that a real API would require." For the export, the size comes from the tenant's most recent `historical_shipments` row. If no size is on record the field is blank. The `no_filter_size` review flag surfaces these before export, but dismiss is allowed because the spec says size issues should not block processing.

### Export gates

Two conditions can intercept an export and show a confirmation dialog:

1. **Pending manual review items** (checked first). Unresolved rows may have incorrect tenant links or missing filter sizes.
2. **No ShipStation data imported** (checked second). Without an import, tenants who already received a filter won't be excluded from the eligible count.

Both are overridable. The review-items check fires first so the more critical data-quality warning can't be bypassed by triggering only the import warning. The gate uses `import_log` rather than inferring from ship dates (see Data Model).

### Export commit before CSV delivery

The `shipments` INSERT is committed before the CSV is streamed. A dismissed download leaves tenants marked as ordered with no CSV delivered. The `export_batches` re-download link recovers the file without creating new records.

## Potential Improvements

- **Synchronous import.** `importService.js` processes every row in one blocking call. Fine at around 195 rows, but it would stall the event loop at 10,000+. The fix is a worker thread or an async SQLite driver.
- **List views.** The eligible tenants list loads fully into client memory, and import history is capped at 50 rows. At scale this would need server-side pagination for import history and virtual scrolling (`react-window`) or a name search for the eligible list.
- **Tracking number write-back.** Tracking numbers come back from ShipStation after fulfillment, and right now there is no mechanism to write them back to `shipments`. Last-shipped dates for tenants exported by this tool will lag by one import cycle until the next ShipStation import.
- **Automated ShipStation sync.** Manual CSV upload is a stand-in for a real API integration.
- **Fuzzy name matching.** Current matching requires an exact name match (case-insensitive). A Levenshtein threshold would catch typos at the cost of producing more `ambiguous_match` flags.
- **Multi-property support.** Dropping the `UNIQUE tenant_id` constraint on `tenant_property` is the only schema change needed. The eligibility query already handles multiple property rows per tenant correctly because `recently_shipped` checks each interval independently, and UNION means a tenant is blocked if they are within any interval, which is equivalent to using the longest one.
