# Air Filter Shipment System

## Setup

**Prerequisites:** Node.js, `sqlite3` CLI on PATH.

```bash
npm install
bash reset.sh        # builds DB, seeds fixtures and properties
```

Then in two terminals:

```bash
npm run dev:backend  # Express on :3000
npm run dev:frontend # Vite on :5173 — open http://localhost:5173
```

> `reset.sh` is destructive — it drops and rebuilds `database.db` from scratch. Run it once on first setup or when you need a clean slate.

## Data Model

The system extends the pre-existing `tenants`, `enrollments`, and `historical_shipments` tables with five new tables.

### `tenant_property`

Maps each tenant to a property so the eligibility engine can look up the shipment interval. The table is shaped like a junction (many-to-many) but carries a `UNIQUE tenant_id` constraint, making it effectively one-to-one in the current implementation — first-insert wins at seed time. The junction shape was chosen to avoid altering the immutable `tenants` table and to leave the door open for multi-property support later without a schema migration. See **Known Limitations** for the tradeoff this creates around the longest-interval rule.

### `shipments`

Records shipments *generated by this tool*. Kept entirely separate from `historical_shipments` (which holds ShipStation-confirmed past deliveries) so the double-ship guard is unambiguous: the eligibility query checks both tables for recent activity, but provenance is never mixed. A row in `shipments` means "we ordered this"; a row in `historical_shipments` means "ShipStation confirmed it shipped." `tracking_number` is nullable — tracking numbers come back from ShipStation after fulfillment, not at order time.

### `historical_shipments` (extended)

Two columns added via `ALTER TABLE` (non-destructive, no existing data affected):

- **`custom_field_1`** — filter size, populated by ShipStation CSV import. Used as `last_filter_size` on the eligibility list and written into the export CSV.
- **`review_reason`** — nullable. `NULL` means the row is clean; a non-null value (`'unmatched'`, `'ambiguous_match'`, `'no_filter_size'`) puts it in the manual review queue. Cleared back to `NULL` when the operator resolves or dismisses it.

This approach avoids a separate queue table and keeps resolution in-place — the shipment record itself is the queue entry. The tradeoff is that import-state bleeds into the shipment record; a larger system might prefer a separate `review_queue` table referencing `historical_shipments` by ID. See **Potential Improvements**.

### `import_log`

Records each actual CSV upload (timestamp, row count, matched count, unresolved count). This exists to power the export gate — the gate checks whether any real import has occurred, not whether ship-date data exists. The seeded fixture data carries historical ship dates, so inferring "has imported" from `MAX(ship_date)` would always return true and the gate would never fire. `import_log` makes the signal explicit.

### `export_batches`

Stores the full generated CSV alongside metadata (date, `as_of`, tenant count). Exists because the export commits `shipments` rows *before* the CSV file is delivered to the browser — if Doug closes the download dialog, those tenants are already marked as ordered and the CSV would otherwise be unrecoverable. The re-download link in the UI serves `GET /api/export/last` without creating new shipment records.

## Assumptions & Decisions

### Eligibility

- **"Today" is 2026-04-24.** Any `ship_date <= 2026-04-24` counts as a received shipment and resets the interval clock. Future-dated records are ignored. Exported-but-unconfirmed shipments in `shipments` also count — an order is enough to disqualify a tenant.
- **Air-filter rider matching** uses `LIKE '%Airfilters Delivery%'` against the `riders` text blob. This catches both `"Free Airfilters Delivery"` and `"Airfilters Delivery ($4)"`. The `riders` column stores a PostgreSQL array-style string (`{val1,val2}`) from the original schema — it isn't normalized, so LIKE is the only practical option without a schema change.
- **Tenants with no property assignment** (36 in the DB) are excluded — no property means no interval to evaluate against.

### `enrollments.active` mixed types

The `active` column stores both integer `false` and string `'true'` across different fixture rows. The eligibility query checks `(e.active = 1 OR e.active = 'true')` to handle both. Normalizing this to `0`/`1` integers would be a straightforward schema migration.

### Properties Seeding

Seeding runs as a separate step (`npm run seed-properties`) after fixtures, not baked into the migration. Reason: `prop-maple-loop` references tenants 900001/900002, which only exist after `seed-fixtures` runs. `INSERT OR IGNORE` keeps the script idempotent.

| Issue in `properties.json` | Decision |
|---|---|
| `prop-riverbend` appears twice (90-day "Riverbend" + 45-day "Riverbend Annex") | First-wins on the property primary key. Annex skipped; tenant 145566 stays on 90-day interval. |
| `prop-oak-overflow` lists tenant 135452, already in `prop-oak-terrace` | `UNIQUE tenant_id` means the second insert is silently ignored — 135452 stays on Oak Terrace. |
| Tenant IDs not present in `tenants` | Skipped with a warning. Only affects `prop-maple-loop` if fixtures haven't run. |

### ShipStation Import Matching & Manual Review Criteria

Automatic match requires both name and zip to agree (case-insensitive, trimmed). Either alone is not treated as confident — address reuse across tenants and name typos are both common enough to warrant manual review over a bad auto-match.

**A row is flagged for manual review under three conditions:**

| Reason | What it means | Required action |
|---|---|---|
| `unmatched` | No tenant found with the same name + zip | Pick the correct tenant from the suggested candidates, or dismiss |
| `ambiguous_match` | 2+ tenants share the same name and zip — auto-select is not safe | Confirm which tenant this shipment belongs to, or dismiss if uncertain |
| `no_filter_size` | Tenant matched confidently, but `custom_field_1` is blank in the CSV | Enter the filter size. **Dismiss is blocked until a size is saved** — a blank size would silently produce a bad export row that ShipStation can't act on |

**What does not get flagged:** a row with a single confident name+zip match and a non-empty filter size is auto-linked and never surfaces to the queue.

**Zip normalization:** ShipStation exports omit leading zeros (e.g. `3095` instead of `03095`). `normalizeZip()` in `importService.js` strips non-digits and left-pads to 5 characters; the SQL compares against `substr(zip, 1, 5)` to handle ZIP+4 suffixes on the tenant side.

### Export gates

Two conditions intercept the export with a popup confirmation dialog before proceeding:

1. **Pending manual review items** — checked first. Exporting with unresolved queue items may produce rows with incorrect tenant links or missing filter sizes.
2. **No ShipStation data imported** — checked second. Without an import, tenants who already received a filter won't be excluded from the eligible list.

Both are overridable ("Export anyway"). The review-items check fires first so operators can't bypass the more critical data-quality warning by seeing only the import warning.

### Import-before-export ordering dependency

The eligibility engine's accuracy depends on `historical_shipments` being current. A fresh deployment with no ShipStation data imported will show an inflated eligible count.

**Mitigation:** The export gate warns when `import_log` is empty. The longer-term fix is automated ShipStation sync rather than manual CSV upload, which eliminates the ordering problem entirely.

### Export commit before CSV delivery

The `shipments` INSERT is committed before the CSV is streamed. A dismissed download or server crash mid-response leaves tenants marked as ordered with no CSV delivered. The `export_batches` re-download link recovers the CSV without creating new records. The UI also resets `eligibilityChecked` after each export, forcing a fresh check before the next batch.

### Data State

`schema.sql` contains 1,258 commented-out `INSERT INTO enrollments` rows — the apparent original dataset. The current DB has only the enrollments from `supplemental-fixtures.sql`, so most tenants are ineligible under the enrollment check. If the original `database.db` was overwritten by `reset.sh`, those enrollments were lost. The eligibility engine is correct against the current schema; restoring the original data would surface the full eligible set.

## Known Limitations

### Multi-property tenants not fully supported

`tenant_property` enforces `UNIQUE tenant_id`, so only one property per tenant is stored. The eligibility query's `recently_shipped` CTE already supports multiple rows per tenant (it uses UNION and DISTINCT), but the outer query is a plain JOIN — removing the UNIQUE constraint without also adding a `MAX(shipment_interval_days)` aggregation per tenant would cause a tenant with two assignments to appear multiple times in results. Both changes need to happen together: drop the UNIQUE constraint, add interval aggregation. The junction table shape was chosen specifically so this is a query change, not a schema redesign.

### Synchronous import blocks event loop

`importService.js` uses better-sqlite3 (synchronous) and processes every row in one blocking call. At ~195 rows this takes under 5ms. A file with 10,000+ rows would block Node's event loop for hundreds of milliseconds, stalling any concurrent requests. Fix: move import processing to a worker thread, or switch to an async SQLite driver.

### Import history capped at 50 rows

`GET /api/imports/recent` returns only the 50 most recent matched rows. The underlying data is complete — this is a display cap. For large deployments where thousands of import records accumulate, the history panel silently truncates. See **Potential Improvements**.

### Tracking number sync gap

Tracking numbers return from ShipStation after fulfillment. There's no mechanism to write them back to `shipments`. As a result, "last shipped" dates for tenants exported by this tool (rather than confirmed via ShipStation import) lag by one import cycle.

### No authentication

Single-operator tool with no session management. Adding auth requires a login flow and session/JWT handling, which was out of scope.

## Potential Improvements

### List views: pagination, virtualization, search

The import history panel is capped at 50 rows; the eligible tenants list loads entirely into client memory. At scale:

- **Server-side pagination** with `LIMIT`/`OFFSET` and a "load more" control covers the import history case with minimal frontend change.
- **Client-side virtual scrolling** (e.g. `react-window`) renders only visible rows — useful when the full list is needed for client-side filtering and stays under ~10,000 rows. Good fit for the eligible tenants panel.
- **Name search** — a text input that filters client-side for small lists, or drives a debounced server query for large ones. Particularly useful when ops is looking for a specific tenant in a large eligible list.

### Automated ShipStation sync

Manual CSV upload is a stand-in for an automated sync. ShipStation's API or webhooks could push confirmed shipments into `historical_shipments` on a schedule, eliminating the import-before-export ordering dependency and the manual upload step.

### Fuzzy name matching

Current matching requires exact first+last name (case-insensitive). A Levenshtein distance threshold would catch common typos (`Mckenzee` → `McKenzie`) at the cost of more `ambiguous_match` flags surfacing in the queue. Useful if ShipStation data has known transcription patterns.

### Multi-property interval tie-break

Once `tenant_property`'s UNIQUE constraint is relaxed (see Known Limitations), the `recently_shipped` CTE needs `MAX(p.shipment_interval_days)` per tenant rather than a per-row JOIN. The outer eligibility query would then `SELECT DISTINCT t.id` to collapse multiple property rows per tenant correctly.

### Export batching by property

No cap exists on the number of eligible tenants per export. A large portfolio could generate a very large ShipStation import file. Adding optional property-level filtering (export only tenants from a selected property) would make batching manageable and reduce the risk of a single oversized upload.

### `review_reason` as a separate queue table

The current design co-mingles import state with the shipment record. A separate `review_queue` table referencing `historical_shipments` by ID would be cleaner for querying, allow multiple review reasons per row if needed, and preserve an audit trail of what was reviewed and when. Tradeoff: an extra JOIN on every queue-related query and slightly more complex resolution logic.
