# Air Filter Shipment System

## Setup

**Prerequisites:** Node.js, `sqlite3` CLI on PATH.

```bash
npm install
bash reset.sh        # builds DB, seeds fixtures and properties
```

Then in two terminals:

```bash
npm run dev:backend  # Express on :3000
npm run dev:frontend # Vite on :5173 — open http://localhost:5173
```

> `reset.sh` is destructive — it drops and rebuilds `database.db` from scratch. Run it once on first setup or when you need a clean slate.

## Data Model

The system extends the pre-existing `tenants`, `enrollments`, and `historical_shipments` tables with new tables, and adds two columns to `historical_shipments`.

### `properties`

One row per property from `properties.json`. `id` is a text slug (e.g. `"prop-cedar-court"`). The key field is `shipment_interval_days` — how many days must pass between shipments for tenants at that property. `address` is nullable; `properties.json` provides none.

### `tenant_property`

Maps each tenant to a property so the eligibility engine can look up the shipment interval. The spec defines a tenant as belonging to one property, so the table carries a `UNIQUE tenant_id` constraint enforcing that. The junction shape (rather than a direct foreign key on `tenants`) was chosen to avoid altering the immutable `tenants` table.

### `shipments`

Records shipments *generated by this tool*. The spec allows using `historical_shipments` directly or building a separate model — a separate table is the right call here because `historical_shipments.tracking_number` is `NOT NULL UNIQUE` in the original schema, and this tool generates export records before tracking numbers exist. Merging would require either relaxing that constraint or inserting placeholder values, both worse than a clean separate table.

Keeping them separate also makes the double-ship guard unambiguous: the eligibility query UNIONs both tables in `recently_shipped` — a tenant is blocked by either a confirmed past delivery OR a pending export order. `tracking_number` is nullable — tracking comes back from ShipStation after fulfillment.

### `historical_shipments` (extended)

- **`custom_field_1`** — filter size, populated by ShipStation CSV import. Used as `last_filter_size` on the eligibility list and written into the export CSV.
- **`review_reason`** — nullable. `NULL` means the row is clean; a non-null value (`'unmatched'`, `'ambiguous_match'`, `'no_filter_size'`) puts it in the manual review queue. Cleared back to `NULL` when the operator resolves or dismisses it. This keeps the queue as a filtered view of `historical_shipments` rather than a separate table — simpler queries, but import state bleeds into the shipment record.

### `import_log`

Records each actual CSV upload. The export gate checks whether any real import has occurred — the seeded fixture data carries historical ship dates, so inferring "has imported" from `MAX(ship_date)` would always return true and the gate would never fire. `import_log` makes the signal explicit.

### `export_batches`

Stores the full generated CSV. Exists because the export commits `shipments` rows *before* the CSV is delivered to the browser — if Doug closes the download dialog, those tenants are already marked as ordered. The re-download link serves `GET /api/export/last` without creating new shipment records.

## Assumptions & Decisions

### Eligibility

- **"Today" is 2026-04-24.** Shipments within a tenant's property interval and on or before this date count toward the cooldown. Future-dated records are ignored. Tenants who have never received a shipment are eligible.
- **Exported-but-unconfirmed shipments count.** The spec asks for a decision here: rows in `shipments` block re-export within the interval window. An order is enough to disqualify — safer default to prevent double-shipping.
- **Air-filter rider matching** uses `LIKE '%Airfilters Delivery%'` against the `riders` text blob. This catches both `"Free Airfilters Delivery"` and `"Airfilters Delivery ($4)"`. The `riders` column is a PostgreSQL array-style string from the original schema — not normalized, so LIKE is the only practical option without a schema change.
- **Tenants with no property assignment** (36 in the DB) are excluded — no property means no interval to evaluate against.

### Loading `properties.json`

`properties.json` is the source of property intervals and tenant assignments per the spec. The file has a few conflicts that need a defined fallback:

| Conflict | Decision |
|---|---|
| Two entries share the same property ID | First entry wins; second is skipped with a warning. |
| One tenant is listed under two different properties | First assignment wins; second is skipped with a warning. |
| Tenant IDs that don't exist in the `tenants` table | Skipped with a warning — no link created. |

In all three cases the script logs a warning rather than failing, so setup completes with the remaining clean data intact.

### ShipStation Import Matching & Manual Review

The spec says to match on name, address, or a combination. We use **name + zip** as the match key. Full address matching is brittle (unit formatting varies, abbreviations differ); zip is the most stable, structured part of the address. Either field alone is not treated as confident.

**A row is flagged for manual review under three conditions:**

| Reason | What it means | Required action |
|---|---|---|
| `unmatched` | No tenant found with the same name + zip | Pick the correct tenant from the suggested candidates, or dismiss |
| `ambiguous_match` | 2+ tenants share the same name and zip | Confirm which tenant this shipment belongs to, or dismiss |
| `no_filter_size` | Matched confidently, but `custom_field_1` is blank | Enter the filter size — dismiss is blocked until saved |

**Zip normalization:** ShipStation exports omit leading zeros (e.g. `3095` instead of `03095`). `normalizeZip()` strips non-digits and left-pads to 5 characters; SQL compares against `substr(zip, 1, 5)` to handle ZIP+4 suffixes.

### Filter sizes

`custom_field_1` cells may contain multiple space-separated sizes (e.g. `"14x20x1 20x20x1"`). We store the raw string verbatim — the spec notes sizes "may lack the structure that a real API would require" and storing verbatim passes the value through to the export CSV unchanged. For the export, `custom_field_1` comes from the tenant's most recent `historical_shipments` row as of the export date. If no size is on record the field is blank — the `no_filter_size` review flag exists to catch this before export.

### Export gates

Two conditions intercept export with a popup confirmation dialog:

1. **Pending manual review items** — checked first. Unresolved rows may have incorrect tenant links or missing filter sizes.
2. **No ShipStation data imported** — checked second. Without an import, tenants who already received a filter won't be excluded from the eligible count.

Both are overridable. The review-items check fires first so operators can't bypass the more critical data-quality warning by seeing only the import warning. The gate uses `import_log` rather than inferring from ship dates — see Data Model.

### Export commit before CSV delivery

The `shipments` INSERT is committed before the CSV is streamed. A dismissed download leaves tenants marked as ordered with no CSV delivered. The `export_batches` re-download link recovers the file without creating new records.

## Known Limitations

- **Synchronous import** — `importService.js` processes every row in one blocking call. Fine at ~195 rows; would stall the event loop on 10,000+. Fix: worker thread or async SQLite driver.
- **Import history capped at 50 rows** — display cap only; underlying data is complete. Would need pagination at scale.
- **Tracking number sync gap** — tracking numbers return from ShipStation after fulfillment with no write-back to `shipments`. "Last shipped" dates for tenants exported by this tool lag by one import cycle.

## Potential Improvements

- **List views** — eligible tenants list loads fully into client memory; import history is capped at 50. At scale: server-side pagination for import history, virtual scrolling (`react-window`) or name search for the eligible list.
- **Automated ShipStation sync** — manual CSV upload is a stand-in. ShipStation webhooks could push confirmed shipments automatically, eliminating the import-before-export dependency.
- **Fuzzy name matching** — current matching requires exact name (case-insensitive). A Levenshtein threshold would catch typos at the cost of more `ambiguous_match` flags.
- **Multi-property support** — only the `UNIQUE tenant_id` constraint on `tenant_property` needs to be dropped. The eligibility query already handles multiple property rows per tenant correctly: `recently_shipped` checks each interval independently, and UNION means a tenant is blocked if they're within any interval — equivalent to using the longest one.
